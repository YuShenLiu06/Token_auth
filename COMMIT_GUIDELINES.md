# Git 提交规范

## 提交信息格式

所有提交信息必须遵循以下格式：

```
vX.X.X
1.功能描述
2.功能描述
3.功能描述（可选）
```

## 格式说明

- **版本号**：使用 `v` 前缀，后跟主版本号.次版本号.修订版本号（如 `v1.1.5`）
- **换行**：版本号后必须换行，不能有空格
- **功能列表**：每个功能点以数字序号开头，后跟英文句点，然后是功能描述
- **功能描述**：简洁明了地描述本次提交的主要功能或修改内容

## 示例

### 正确示例
```
v1.1.5
1.更新README文档，添加新功能说明
2.创建v1.1.5版本更新日志文件
```

```
v1.1.4
1.优化了命令结构
2.新增了SSL开关
```

### 错误示例
```
v1.1.5 1.更新README文档，添加新功能说明 2.创建v1.1.5版本更新日志文件
```
（错误：版本号后有空格而不是换行）

```
更新README文档
```
（错误：缺少版本号和格式）

## 注意事项

1. 每次提交都应该有明确的版本号
2. 功能描述应该简洁明了，避免过于冗长
3. 如果是修复bug，可以使用"修复"作为描述开头
4. 如果是新增功能，可以使用"新增"或"添加"作为描述开头
5. 如果是优化或改进，可以使用"优化"作为描述开头

## 版本号规则

- **主版本号**：不兼容的API修改
- **次版本号**：向下兼容的功能性新增
- **修订版本号**：向下兼容的问题修正

例如：
- `v1.0.0` → `v1.1.0`：新增功能
- `v1.1.0` → `v1.1.1`：修复bug
- `v1.1.1` → `v2.0.0`：重大更新，不兼容旧版本
